# GitベースCMS構成候補（比較用）

## 目的
- `evaluation-criteria.md` の評価軸で、GitベースCMS構成候補を比較する。

## 候補一覧（GitベースCMS）

| 構成 | 区分 | 向いているケース | メモ |
|---|---|---|---|
| Astro + Decap CMS | SSG + Git連携CMS | 静的配信を維持しつつ編集画面も欲しい | 実務での導入しやすさと拡張のバランスがよい。 |
| Next.js + TinaCMS | React + Git連携CMS | React前提でコンテンツ編集体験を作り込みたい | 表現力は高いが運用設計が必要。 |
| Hugo + CloudCannon | SSG + SaaS編集 | 高速配信と編集導線を両立したい | ビルド速度と運用分離がしやすい。 |
| Jekyll + Decap CMS | SSG + Git連携CMS | 小規模サイトを低コストで継続運用 | シンプルだが表現自由度は限定される。 |

## 先に比較に載せる推奨候補

1. Astro + Decap CMS
2. Hugo + CloudCannon
3. Next.js + TinaCMS
4. Jekyll + Decap CMS

## 評価前提

- スコア: `5（非常に良い）- 1（弱い）`
- 想定: ポップカルチャー紹介のブランドメディアを、少人数で継続運用するケース。

## 候補別 採点理由（観点別）

### Astro + Decap CMS
- 運用継続性 `3`: Git運用とCMS運用の両立設計が必要。
- 引き継ぎ容易性 `3`: 手順化すれば引き継げるが学習コストはある。
- 編集体験 `3`: CMS画面はあるが高度な編集は工夫が必要。
- 表現自由度 `4`: Astroで十分な表現を作り込める。
- 表示性能 `5`: 静的配信で高速。
- SEO対応力 `3`: 実装で対応可能だが運用品質に依存。
- セキュリティ運用 `4`: 動的面が少なく管理しやすい。
- 拡張性 `3`: 要件追加時は実装対応が必要。
- プラグイン/拡張エコシステム `3`: 最低限の拡張資産はある。
- コスト（初期） `3`: 初期構築は中程度。
- コスト（運用） `4`: 配信費用は抑えやすい。
- 日本での使いやすさ `2`: 国内事例は多くない。
- 採用/外注しやすさ `2`: 実務経験者は限定的。
- ベンダーロックイン `4`: Git資産中心で移行しやすい。

### Next.js + TinaCMS
- 運用継続性 `3`: 実装自由度が高い分、運用設計が必要。
- 引き継ぎ容易性 `2`: 独自実装依存が強いと引き継ぎ難度が上がる。
- 編集体験 `4`: 編集体験を作り込める。
- 表現自由度 `5`: Reactで高い表現力を持てる。
- 表示性能 `4`: 最適化次第で高速化できる。
- SEO対応力 `4`: フロント実装で細かく制御しやすい。
- セキュリティ運用 `3`: 保守範囲が広くなる。
- 拡張性 `4`: 機能拡張に対応しやすい。
- プラグイン/拡張エコシステム `3`: 周辺資産はあるが統合は自力。
- コスト（初期） `2`: 初期実装工数が高い。
- コスト（運用） `3`: 開発保守が継続する。
- 日本での使いやすさ `2`: 国内情報は限定的。
- 採用/外注しやすさ `2`: 実務経験者は少なめ。
- ベンダーロックイン `3`: 構成依存はあるが移行余地はある。

### Hugo + CloudCannon
- 運用継続性 `4`: ビルド速度が高く運用が安定しやすい。
- 引き継ぎ容易性 `4`: 運用ルールを整えると再現しやすい。
- 編集体験 `4`: SaaS編集で非エンジニア運用を補える。
- 表現自由度 `3`: 表現は可能だがReact系より制約がある。
- 表示性能 `5`: 静的配信で高性能。
- SEO対応力 `3`: 基本対応は可能で、設計次第。
- セキュリティ運用 `4`: 保守対象が少なく安全運用しやすい。
- 拡張性 `3`: 複雑要件には追加設計が必要。
- プラグイン/拡張エコシステム `3`: 必要十分の拡張資産。
- コスト（初期） `3`: 初期準備は中程度。
- コスト（運用） `3`: SaaS費用と保守工数のバランス型。
- 日本での使いやすさ `2`: 国内情報は少ない。
- 採用/外注しやすさ `2`: 対応可能な外注先は限定的。
- ベンダーロックイン `3`: SaaS依存分の移行負荷はある。

### Jekyll + Decap CMS
- 運用継続性 `3`: 小規模なら継続しやすい。
- 引き継ぎ容易性 `3`: 構成は単純だがGit理解は必要。
- 編集体験 `3`: 最低限の編集導線は確保可能。
- 表現自由度 `2`: 表現面は限定的。
- 表示性能 `4`: 静的配信で十分高速。
- SEO対応力 `3`: 基本対応は可能。
- セキュリティ運用 `4`: シンプル構成で守りやすい。
- 拡張性 `2`: 複雑機能に弱い。
- プラグイン/拡張エコシステム `2`: 最新要件への即応資産は少ない。
- コスト（初期） `4`: 低コストで開始しやすい。
- コスト（運用） `4`: ランニング低め。
- 日本での使いやすさ `2`: 国内実務情報は少ない。
- 採用/外注しやすさ `2`: 人材母数が少ない。
- ベンダーロックイン `4`: 移行余地は高い。
