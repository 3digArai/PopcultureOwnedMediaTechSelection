# 静的サイト構成候補（比較用）

## 目的
- `evaluation-criteria.md` の評価軸で、静的サイト構成候補を比較する。

## 候補一覧（静的サイト）

| 構成 | 区分 | 向いているケース | メモ |
|---|---|---|---|
| Astro + microCMS | SSG + SaaS CMS | 記事更新を非エンジニア中心で回したい | 静的配信の速さと編集運用の両立がしやすい。 |
| Astro + Markdown（Content Collections） | SSG + Git運用 | 技術チーム中心、小規模〜中規模メディア | 低コストで高速、運用ルールが明確なら強い。 |
| Hugo + Markdown | SSG + Git運用 | 更新頻度が高く、ビルド速度を重視 | 実績が多く、軽量配信に強い。 |
| Eleventy + Decap CMS | SSG + Git連携CMS | Git管理しつつ編集画面も欲しい | 編集とレビュー履歴を両立しやすい。 |
| Next.js（SSG）+ MDX | React SSG | 表現を作り込みたい静的運用 | UI自由度が高く、設計次第で高速化可能。 |
| Jekyll + GitHub Pages | SSG + ホスティング一体 | 小規模サイトを低コストで運用 | シンプル構成で保守しやすい。 |

## 先に比較に載せる推奨候補

1. Astro + microCMS
2. Hugo + Markdown
3. Astro + Markdown（Content Collections）
4. Next.js（SSG）+ MDX
5. Eleventy + Decap CMS
6. Jekyll + GitHub Pages

## 評価前提

- スコア: `5（非常に良い）- 1（弱い）`
- 想定: ポップカルチャー紹介のブランドメディアを、少人数で継続運用するケース。

## 候補別 採点理由（観点別）

### Astro + microCMS
- 運用継続性 `4`: CMSで更新しつつ配信は静的化でき、運用負荷を抑えやすい。
- 引き継ぎ容易性 `4`: CMS操作とAstro構成の役割分担が明確で手順化しやすい。
- 編集体験 `4`: microCMS側の編集UIで非エンジニアも更新しやすい。
- 表現自由度 `4`: Astroで十分なUI表現が可能。
- 表示性能 `5`: 静的配信で高い表示速度を確保しやすい。
- SEO対応力 `4`: 主要なSEO要件を実装で担保しやすい。
- セキュリティ運用 `4`: サーバーの動的面が少なく、リスクを抑えやすい。
- 拡張性 `4`: API経由で機能追加しやすい。
- プラグイン/拡張エコシステム `3`: 拡張手段はあるがCMSプラグイン型より少ない。
- コスト（初期） `4`: 比較的短期間で立ち上げやすい。
- コスト（運用） `4`: 静的配信でランニングを抑えやすい。
- 日本での使いやすさ `4`: 国内情報と運用事例を見つけやすい。
- 採用/外注しやすさ `4`: フロント人材で対応しやすい。
- ベンダーロックイン `3`: CMS依存はあるが、フロント分離で移行余地もある。

### Astro + Markdown（Content Collections）
- 運用継続性 `3`: 運用は安定するが、更新担当にGit運用理解が必要。
- 引き継ぎ容易性 `3`: ルール化すれば引き継げるが、非エンジニアにはハードルがある。
- 編集体験 `2`: 直接Markdown編集が中心で、編集者向けUIは弱い。
- 表現自由度 `4`: Astroのコンポーネント構成で十分に作り込める。
- 表示性能 `5`: 静的配信で高性能を出しやすい。
- SEO対応力 `3`: 実装で対応可能だが、運用者の手入力品質に依存しやすい。
- セキュリティ運用 `5`: 動的機能が少なく、保守対象を絞りやすい。
- 拡張性 `3`: 拡張は可能だが、要件次第で実装負荷が上がる。
- プラグイン/拡張エコシステム `2`: すぐ使える運用機能は少なめ。
- コスト（初期） `5`: 最小構成で低コスト立ち上げが可能。
- コスト（運用） `5`: 配信コストを低く維持しやすい。
- 日本での使いやすさ `3`: 技術者には扱いやすいが、非エンジニア運用には不向き。
- 採用/外注しやすさ `3`: フロント実装者は確保しやすいが、編集運用設計が課題。
- ベンダーロックイン `5`: Markdown資産の可搬性が高い。

### Hugo + Markdown
- 運用継続性 `4`: ビルドが速く、更新頻度が高い運用でも安定しやすい。
- 引き継ぎ容易性 `4`: 構造がシンプルで、ドキュメント化すれば再現しやすい。
- 編集体験 `2`: Markdown編集中心で非エンジニアには不向き。
- 表現自由度 `3`: 実装は可能だが、React系に比べ表現設計はやや制約がある。
- 表示性能 `5`: 静的サイトとして高性能。
- SEO対応力 `3`: 基本対応は可能だが、運用設計次第で品質差が出る。
- セキュリティ運用 `5`: 静的配信で攻撃面を小さくできる。
- 拡張性 `3`: 要件追加時は設計見直しが必要になりやすい。
- プラグイン/拡張エコシステム `3`: テーマや拡張はあるが、即導入機能は限定的。
- コスト（初期） `5`: 小さく始めやすい。
- コスト（運用） `5`: ランニングコストを低く保ちやすい。
- 日本での使いやすさ `3`: 国内情報はあるが、一般編集運用向けではない。
- 採用/外注しやすさ `3`: 対応できるエンジニアは一定数いる。
- ベンダーロックイン `5`: コンテンツ可搬性が高い。

### Eleventy + Decap CMS
- 運用継続性 `3`: Git運用とCMS運用を両立できるが、運用設計が必要。
- 引き継ぎ容易性 `3`: 仕組みを理解すれば回せるが、学習コストはある。
- 編集体験 `3`: CMS画面はあるが、運用導線の設計に依存。
- 表現自由度 `4`: テンプレート設計で柔軟に対応可能。
- 表示性能 `4`: 静的配信で十分高速。
- SEO対応力 `3`: 実装で担保できるが、初期設計の質に依存。
- セキュリティ運用 `4`: 動的面が少なく保守しやすい。
- 拡張性 `3`: 要件追加時に実装コストが増えやすい。
- プラグイン/拡張エコシステム `3`: 最低限の拡張は可能。
- コスト（初期） `3`: 標準実装の準備に中程度の工数が必要。
- コスト（運用） `4`: 配信コストは抑えやすい。
- 日本での使いやすさ `2`: 国内事例・情報が少なめ。
- 採用/外注しやすさ `2`: 国内での実務経験者は多くない。
- ベンダーロックイン `4`: Git資産中心で移行余地がある。

### Next.js（SSG）+ MDX
- 運用継続性 `3`: 実装自由度は高いが、継続保守の技術負荷はある。
- 引き継ぎ容易性 `3`: 設計ドキュメントがないと引き継ぎ難度が上がる。
- 編集体験 `2`: MDX直編集中心だと非エンジニア運用は難しい。
- 表現自由度 `5`: Reactでブランド表現を最大限作り込める。
- 表示性能 `5`: SSGで高速化しやすい。
- SEO対応力 `4`: 実装で細かく制御できる。
- セキュリティ運用 `4`: サイト配信自体は安全にしやすい。
- 拡張性 `4`: 追加機能に対応しやすい。
- プラグイン/拡張エコシステム `3`: React/Next周辺の資産を活用可能。
- コスト（初期） `3`: 設計と実装に一定工数が必要。
- コスト（運用） `3`: 開発保守の継続が必要。
- 日本での使いやすさ `3`: 技術情報は多いが、編集運用向け情報は少ない。
- 採用/外注しやすさ `4`: Next.js人材は比較的確保しやすい。
- ベンダーロックイン `4`: MDX運用で可搬性を保ちやすい。

### Jekyll + GitHub Pages
- 運用継続性 `3`: 構成は簡単だが、運用は技術者前提になりやすい。
- 引き継ぎ容易性 `3`: 仕組みは単純で引き継ぎやすいが、Git理解は必要。
- 編集体験 `2`: 編集画面がなく、執筆担当には不向き。
- 表現自由度 `2`: 表現面の自由度は他候補より低い。
- 表示性能 `4`: 静的配信で十分高速。
- SEO対応力 `3`: 基本対応はできるが高度制御は実装次第。
- セキュリティ運用 `4`: 配信基盤を含めてシンプルに運用しやすい。
- 拡張性 `2`: 複雑要件への拡張は難しい。
- プラグイン/拡張エコシステム `2`: 現代的要件への即応資産は少なめ。
- コスト（初期） `5`: 低コストで開始しやすい。
- コスト（運用） `5`: 低ランニングで維持しやすい。
- 日本での使いやすさ `3`: 情報はあるが実務採用は限定的。
- 採用/外注しやすさ `2`: 対応可能な外注先は絞られる。
- ベンダーロックイン `4`: 静的資産で移行しやすい。
