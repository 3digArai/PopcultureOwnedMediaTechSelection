# Headless構成候補（比較用）

## 目的
- `evaluation-criteria.md` の評価軸で、Headless構成候補を比較する。

## 候補一覧（Headless）

| 構成 | 区分 | 向いているケース | メモ |
|---|---|---|---|
| microCMS + Next.js | SaaS CMS + React | 日本語運用、少人数チーム、記事中心メディア | 国内サポートと日本語UIで運用しやすい。 |
| Contentful + Next.js | SaaS CMS + React | 多言語やグローバル運用 | モデル設計の柔軟性が高い。 |
| Sanity + Next.js | SaaS/自前併用 + React | 表現を作り込みたいブランドサイト | Studioのカスタマイズ性が高い。 |
| Strapi + Next.js | OSS self-host + React | データ主権と拡張を重視 | 自社運用前提で自由度が高い。 |
| Hygraph + Next.js | GraphQL SaaS + React | GraphQL中心の設計で運用したい | API中心でフロント連携しやすい。 |
| WordPress（Headless）+ Next.js | 既存CMS活用 + React | 既存WP編集体験を活かして高速化したい | 編集はWP、配信はNextで分離運用。 |

## 先に比較に載せる推奨候補

1. microCMS + Next.js
2. Sanity + Next.js
3. Contentful + Next.js
4. WordPress（Headless）+ Next.js
5. Strapi + Next.js
6. Hygraph + Next.js

## 評価前提

- スコア: `5（非常に良い）- 1（弱い）`
- 想定: ポップカルチャー紹介のブランドメディアを、少人数で継続運用するケース。

## 候補別 採点理由（観点別）

### microCMS + Next.js
- 運用継続性 `4`: SaaS運用で保守対象を減らしつつ、更新導線を整理しやすい。
- 引き継ぎ容易性 `4`: 日本語情報が多く、運用手順をドキュメント化しやすい。
- 編集体験 `4`: コンテンツ入力画面が分かりやすく、編集担当が使いやすい。
- 表現自由度 `4`: Next側で表現を作り込めるが、CMS側UIの自由度には上限がある。
- 表示性能 `5`: NextのSSG/ISRと相性が良く、高速化しやすい。
- SEO対応力 `4`: メタ/OGP/構造化はフロント実装で十分に制御可能。
- セキュリティ運用 `4`: CMS基盤をSaaSに寄せられるため、運用負荷を抑えやすい。
- 拡張性 `4`: API中心で機能追加しやすいが、大規模要件では設計力が必要。
- プラグイン/拡張エコシステム `3`: 拡張手段はあるが、WordPressほど即導入資産は多くない。
- コスト（初期） `4`: 標準構成で立ち上げやすく、初期構築を進めやすい。
- コスト（運用） `3`: SaaS利用料とフロント保守の両方が継続する。
- 日本での使いやすさ `5`: 国内事例・日本語ドキュメント・支援会社が見つけやすい。
- 採用/外注しやすさ `4`: Next人材と国内CMS知見の組み合わせを確保しやすい。
- ベンダーロックイン `2`: SaaS依存があり、移行時にモデルとAPI再設計が必要になりやすい。

### Contentful + Next.js
- 運用継続性 `4`: 安定したSaaS基盤で継続運用しやすい。
- 引き継ぎ容易性 `3`: 機能は豊富だが、設計ルールがないと属人化しやすい。
- 編集体験 `4`: 編集機能は充実しており、運用体制を作りやすい。
- 表現自由度 `4`: モデル設計の柔軟性が高く、ブランド要件に対応しやすい。
- 表示性能 `5`: Nextとの組み合わせで高速配信しやすい。
- SEO対応力 `4`: フロント側の実装自由度でSEO要件を担保できる。
- セキュリティ運用 `5`: SaaS運用でインフラ保守負荷が低い。
- 拡張性 `4`: API/モデルの柔軟性は高いが、複雑化時に設計難易度は上がる。
- プラグイン/拡張エコシステム `4`: 連携は豊富だが、導入設計が必要。
- コスト（初期） `3`: 設計と初期セットアップに一定工数が必要。
- コスト（運用） `2`: 利用規模に応じてランニングコストが上がりやすい。
- 日本での使いやすさ `3`: 日本語情報はあるが、国内運用ノウハウは限定的。
- 採用/外注しやすさ `3`: 対応人材はいるが、国内市場では母数が多くない。
- ベンダーロックイン `2`: SaaS依存度が高く、移行時の再設計負担が大きい。

### Sanity + Next.js
- 運用継続性 `4`: 基盤運用は軽く、運用フローを整えると回しやすい。
- 引き継ぎ容易性 `3`: Studioの作り込み次第で、引き継ぎ難易度に差が出る。
- 編集体験 `4`: 適切に設計すれば編集UIを要件に合わせて最適化できる。
- 表現自由度 `5`: コンテンツモデルとStudio拡張で高い自由度を持てる。
- 表示性能 `5`: Nextと相性が良く高速配信を実現しやすい。
- SEO対応力 `4`: フロント実装でSEO制御しやすい。
- セキュリティ運用 `4`: SaaS前提で保守対象を絞りやすい。
- 拡張性 `5`: 構造化コンテンツ設計とカスタム拡張に強い。
- プラグイン/拡張エコシステム `4`: エコシステムは豊富だが設計知識が前提。
- コスト（初期） `3`: モデル/Studio設計に初期工数が必要。
- コスト（運用） `3`: SaaS費用と実装保守のバランス運用になる。
- 日本での使いやすさ `2`: 国内の事例・情報量は比較的少ない。
- 採用/外注しやすさ `3`: フロント強い人材で対応できるが、専任経験者は限られる。
- ベンダーロックイン `3`: 移行は可能だが、スキーマと運用設計の移植コストがある。

### Strapi + Next.js
- 運用継続性 `3`: 自社で基盤保守する前提のため、運用体制が必要。
- 引き継ぎ容易性 `3`: OSSで資料はあるが、構築内容の理解が必要。
- 編集体験 `3`: 標準UIは使えるが、運用要件に合わせるには調整が必要。
- 表現自由度 `5`: モデル設計とAPI拡張の自由度が高い。
- 表示性能 `4`: Next側最適化で十分な性能を出せる。
- SEO対応力 `4`: フロント実装で自由に対応できる。
- セキュリティ運用 `3`: 自前運用分のセキュリティ責任が増える。
- 拡張性 `5`: カスタム実装で要件対応しやすい。
- プラグイン/拡張エコシステム `4`: プラグインとコード拡張の両面で対応可能。
- コスト（初期） `2`: 初期構築とインフラ設計に工数がかかりやすい。
- コスト（運用） `3`: SaaS費用は抑えられるが保守工数が継続する。
- 日本での使いやすさ `2`: 国内事例が少なく、運用相談先が限られる。
- 採用/外注しやすさ `3`: Node/Next人材で対応可能だが実績者は限定的。
- ベンダーロックイン `4`: OSSで移行余地は高い。

### Hygraph + Next.js
- 運用継続性 `4`: SaaS運用で継続しやすいが、設計はAPI前提になる。
- 引き継ぎ容易性 `3`: GraphQL前提の理解が必要で、非エンジニア主導は難しい。
- 編集体験 `4`: 編集機能は十分だが、モデル設計の理解が前提。
- 表現自由度 `4`: API中心で柔軟に構築できる。
- 表示性能 `5`: Next連携で高速配信しやすい。
- SEO対応力 `4`: フロント側実装でSEO制御が可能。
- セキュリティ運用 `5`: SaaSで基盤運用負荷を下げやすい。
- 拡張性 `4`: GraphQLベースで拡張しやすいが、運用難易度は上がる。
- プラグイン/拡張エコシステム `3`: 連携はあるが、国内での知見蓄積は限定的。
- コスト（初期） `3`: 設計と実装に一定工数が必要。
- コスト（運用） `2`: 利用規模に応じて費用負担が増えやすい。
- 日本での使いやすさ `2`: 日本語情報・国内事例が少ない。
- 採用/外注しやすさ `2`: 対応できる人材の母数が少なめ。
- ベンダーロックイン `2`: SaaS依存度が高く、移行時の再設計負担が大きい。

### WordPress（Headless）+ Next.js
- 運用継続性 `3`: CMSとフロントの二層管理になるため運用設計が必要。
- 引き継ぎ容易性 `3`: WP側は引き継ぎやすいが、Next側の実装理解が必要。
- 編集体験 `3`: 編集はWPでやりやすいが、プレビューや反映導線に工夫が要る。
- 表現自由度 `5`: フロントを完全に分離でき、表現を作り込みやすい。
- 表示性能 `5`: Next配信で高性能を出しやすい。
- SEO対応力 `5`: フロント実装でSEOを細かく制御しやすい。
- セキュリティ運用 `3`: WP保守とフロント保守の両方が必要。
- 拡張性 `5`: WPのコンテンツ資産を活かしつつフロント機能を拡張できる。
- プラグイン/拡張エコシステム `5`: WP側の拡張資産を利用しやすい。
- コスト（初期） `2`: 通常WPより実装範囲が広く初期負荷が大きい。
- コスト（運用） `3`: 保守対象が2系統のため、工数は中程度以上になりやすい。
- 日本での使いやすさ `4`: WP運用の国内知見を活用しやすい。
- 採用/外注しやすさ `4`: WPとNextの両方を扱える体制なら外注可能性は高い。
- ベンダーロックイン `3`: WP由来の可搬性はあるが、二層構成の移行負担は残る。
