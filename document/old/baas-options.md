# BaaS連携構成候補（比較用）

## 目的
- `evaluation-criteria.md` の評価軸で、BaaS連携構成候補を比較する。

## 候補一覧（BaaS）

| 構成 | 区分 | 向いているケース | メモ |
|---|---|---|---|
| Next.js + Supabase | React + BaaS | 会員機能や投稿機能を段階的に追加したい | SQLベースで拡張しやすい。 |
| Next.js + Firebase | React + BaaS | まず早く作り、将来機能を追加したい | 認証/DB/配信を一体で扱える。 |
| Nuxt + Appwrite | Vue + BaaS | オープンな構成で機能を足したい | 自由度は高いが運用体制が必要。 |
| SvelteKit + PocketBase | SSG/SSR + 軽量BaaS | 小〜中規模を軽快に構築したい | 構成は軽いが国内知見が少ない。 |

## 先に比較に載せる推奨候補

1. Next.js + Supabase
2. Next.js + Firebase
3. Nuxt + Appwrite
4. SvelteKit + PocketBase

## 評価前提

- スコア: `5（非常に良い）- 1（弱い）`
- 想定: ポップカルチャー紹介のブランドメディアを、少人数で継続運用するケース。

## 候補別 採点理由（観点別）

### Next.js + Supabase
- 運用継続性 `3`: 機能は強いが構成管理が必要。
- 引き継ぎ容易性 `3`: DB/認証設計の理解が必要。
- 編集体験 `3`: 編集画面を作り込む前提。
- 表現自由度 `5`: フロント自由度が高い。
- 表示性能 `4`: 最適化次第で高性能化可能。
- SEO対応力 `4`: 実装制御しやすい。
- セキュリティ運用 `3`: 認証/権限設計の責任が増える。
- 拡張性 `5`: 将来機能追加に強い。
- プラグイン/拡張エコシステム `3`: 周辺資産はあるが統合作業が必要。
- コスト（初期） `2`: 初期実装が重くなりやすい。
- コスト（運用） `3`: サービス費用と保守工数の両方が発生。
- 日本での使いやすさ `3`: 情報は増えているが運用ノウハウ設計が必要。
- 採用/外注しやすさ `3`: フルスタック体制が必要。
- ベンダーロックイン `3`: 依存はあるが移行余地は残る。

### Next.js + Firebase
- 運用継続性 `3`: 機能連携はしやすいが運用設計は必要。
- 引き継ぎ容易性 `3`: Firebase特有設計の理解が必要。
- 編集体験 `3`: 管理画面品質は実装次第。
- 表現自由度 `4`: ブランド表現に対応可能。
- 表示性能 `4`: 構成次第で良好。
- SEO対応力 `3`: 実装で対応可能。
- セキュリティ運用 `3`: ルール設定と権限管理の継続運用が必要。
- 拡張性 `4`: 機能追加しやすい。
- プラグイン/拡張エコシステム `3`: 連携資産は一定ある。
- コスト（初期） `3`: 初期は中程度。
- コスト（運用） `3`: 使用量に応じて費用変動。
- 日本での使いやすさ `3`: 国内情報は比較的多い。
- 採用/外注しやすさ `3`: 対応人材は一定数いる。
- ベンダーロックイン `2`: プラットフォーム依存が強い。

### Nuxt + Appwrite
- 運用継続性 `3`: 自由度は高いが運用体制が必要。
- 引き継ぎ容易性 `2`: 独自構成の理解が必要。
- 編集体験 `3`: 管理機能は設計次第。
- 表現自由度 `4`: 十分な表現力を確保できる。
- 表示性能 `4`: 構成次第で高速化できる。
- SEO対応力 `3`: 実装で対応可能。
- セキュリティ運用 `2`: 自前運用部分の責務が増える。
- 拡張性 `4`: 要件追加に対応しやすい。
- プラグイン/拡張エコシステム `3`: 最低限の拡張は可能。
- コスト（初期） `2`: 初期実装は重め。
- コスト（運用） `3`: 保守工数が必要。
- 日本での使いやすさ `2`: 国内事例は少ない。
- 採用/外注しやすさ `2`: 対応人材は限定的。
- ベンダーロックイン `4`: オープン構成で移行余地は高い。

### SvelteKit + PocketBase
- 運用継続性 `3`: 軽量だが運用ルール整備が必要。
- 引き継ぎ容易性 `2`: 対応経験者が少ない。
- 編集体験 `3`: 管理導線は設計依存。
- 表現自由度 `3`: 必要な表現は可能。
- 表示性能 `4`: 軽量に保ちやすい。
- SEO対応力 `3`: 実装で担保可能。
- セキュリティ運用 `2`: 自前運用要素の管理が必要。
- 拡張性 `3`: 中規模までなら対応しやすい。
- プラグイン/拡張エコシステム `2`: 実務向け資産は少ない。
- コスト（初期） `3`: 小規模開始はしやすい。
- コスト（運用） `3`: 継続保守が必要。
- 日本での使いやすさ `2`: 国内情報が少ない。
- 採用/外注しやすさ `2`: 人材母数が少ない。
- ベンダーロックイン `4`: 可搬性は比較的高い。
