# オウンドメディア構築まとめ（現時点）

## 前提
- ポップカルチャー紹介サイトを作る
- 1年で独自メディア/ブランドを立ち上げる
- 初期は飯塚さんが執筆を担当する
- 記事執筆負荷を下げるため、外部リンク紹介記事も併用する

## 結論
- まずは `WordPress単体` で進めるのが最も現実的
- 必要に応じて段階的に拡張する

## 実装方式別まとめ（工数 + 継続費用）
- 前提:
  - 継続費用は月額の目安（初期開発費は別）
  - 為替換算は `1 USD = 155.83円`
  - WordPress.com価格は公式ページの表示（2026-03-05確認）を使用

### 1) WordPress.com単体（ホスティング込み）
- 工数: 低（最短で開始可能）
- 継続費用目安:
  - 年払い基準: Personal `$4/月（約623円/月）`、Premium `$8/月（約1,247円/月）`、Business `$25/月（約3,896円/月）`、Commerce `$45/月（約7,012円/月）`
  - 月払い基準: Personal `$9/月（約1,402円/月）`、Premium `$18/月（約2,805円/月）`、Business `$40/月（約6,233円/月）`、Commerce `$70/月（約10,908円/月）`
- コメント:
  - サーバー運用不要で初速は最も高い
  - 高度カスタムは上位プラン依存

### 2) WordPress.org + AWS Lightsail（テンプレ運用）
- 工数: 低（エンジニアなら半日〜2日で初期立ち上げ）
- 継続費用目安:
  - `$6/月（約935円/月）` から
  - 内訳例: WPインスタンス `$5` + Object Storage `$1`
- コメント:
  - 低コストで開始しやすく、将来拡張も可能
  - サーバー最低限の運用は必要

### 3) WordPress.org + AWS（EC2/RDS/S3/CloudFront）
- 工数: 中（構成設計・運用設計が必要）
- 継続費用目安:
  - 小規模: `120〜220 USD/月（約18,700〜34,300円/月）`
  - 中規模: `250〜450 USD/月（約39,000〜70,100円/月）`
  - 拡張: `500〜900 USD/月（約77,900〜140,200円/月）`
- コメント:
  - 可用性・拡張性は高い
  - 運用人件費が増えやすい

### 4) microCMS + フロント実装（Next.js/Astro）
- 工数: 中
- 継続費用目安:
  - microCMS: Hobby `0円` / Team `4,900円〜` / Business `75,000円〜`
  - フロント配信: `約$5/月（約780円）〜`（小規模想定）
- コメント:
  - 表示自由度は高い
  - プレビュー/再ビルド運用設計が必要

### 5) 管理画面 + フロント自作（自作CMS）
- 工数: 高
- 継続費用目安:
  - インフラ費 + 保守運用人件費（最も高くなりやすい）
- コメント:
  - 自由度は最大
  - 引き継ぎ/運用設計まで含めると負荷が重い

## 実装方式（上から順に現実的）

## WordPress.com単体で開始（ホスティング込み）
- 例: WordPress.comのテーマを選び、管理画面中心で公開運用
- 工数: 低
- 管理コスト: 低
- 特徴: サーバー運用を持たずに最短で開始できる
- 注意: プランによりテーマ/プラグイン/カスタム可能範囲が変わる

## WordPressテンプレートを選んでAWSでホスティング
- 例: 既存テーマ（テンプレート） + 最小調整 + AWS Lightsailで公開
- 工数: 低
- 管理コスト: 低〜中
- 特徴: 立ち上げ最速。将来は子テーマ/独自実装へ段階拡張しやすい

## WordPressを基本にして、フロントをコーディング
- 例: WordPress編集 + テーマ/子テーマで公開画面を実装
- 工数: 低〜中
- 管理コスト: 低〜中
- 特徴: 運用機能を再実装せず、見た目を強化できる

## microCMSなどでAPI定義を作り、フロント+プレビューをコーディング
- 例: Headless CMS + Next.js/Astro
- 工数: 中
- 管理コスト: 中
- 特徴: 表示自由度は上がるが、プレビュー/再ビルド設計が必要

## 管理画面+プレビュー+フロントを自作コーディング
- 例: 自作CMS（管理UI/認証/権限を含む）
- 工数: 高
- 管理コスト: 高
- 特徴: 自由度最大だが、保守責任と運用負荷も最大

## 段階比較の要点
- 段階が上がるほどカスタマイズ性は上がる
- 同時に開発工数と管理コストも上がる
- 今回は「1年で継続運用を成立させること」を優先し、最初はWordPress単体を採用する

## 継続費用の目安（2026-03-02時点）
- 前提: ここでは「毎月の継続費用」を比較する。初期開発費は含めない。
- 前提: `microCMS` はCMS基盤（管理画面/API/メディア）をホストするサービスで、公開Webフロント自体は別途ホスティングが必要。
- 前提: AWS費用は公式のLightsail Pricing Exampleを基準にした概算。
- 為替換算: 1 USD = `155.83円`（FRED `DEXJPUS` の最新30営業日平均: 2026-01-08〜2026-02-20）

### 0) WordPress.com単体で運用する場合
- 構成: CMS + ホスティング一体型（サーバー運用不要）
- 費用構造:
  - プラン課金（無料プラン〜上位プラン）
  - プラン上位ほどテーマ/プラグイン/カスタム機能が解放される
- 目安の見方:
  - 最小開始コストは低い
  - 本格的なカスタム運用をすると上位プランが必要になりやすい
- 補足:
  - 将来 `WordPress.org` へ移行する場合、記事移行はしやすいが、デザイン完全一致は難しい
  - 「近い見た目に再構築する」前提で計画する

### 1) WordPressだけで運用する場合（WordPress単体）
- WordPress本体: `無料（OSS）`
- サーバー目安（AWS Lightsail 例）:
  - WordPressインスタンス `$5/月（約780円/月）`
  - Object Storage `$1/月（約156円/月）`
  - 合計 `約$6/月（約935円/月）`（CDNは初年度無料枠あり）
- 補足: プラグイン有料版、テーマ有料版、外部バックアップ/SaaS監視を使うと別途上乗せ。

#### WordPressをAWSでホスティングする場合（実務上の選択肢）
- 小規模で早く始める: `AWS Lightsail + WordPress Blueprint`
  - サーバー管理の難易度が低く、最短で公開しやすい。
  - 目安費用は上記の `約$6/月（約935円/月）` から。
- 中規模以上・拡張前提: `EC2 + RDS + S3 + CloudFront`
  - 可用性や拡張性は高いが、構成設計と運用設計が必要。
  - 費用はトラフィック・冗長化要件で変動しやすく、Lightsailより高くなるのが一般的。
- 今回の方針:
  - 初期は `Lightsail` で開始し、アクセス増や要件増加時に `EC2/RDS構成` へ段階移行する。

#### `EC2 + RDS + S3 + CloudFront` の月額目安（概算レンジ）
- 前提:
  - 為替は本書の前提と同じ `1 USD = 155.83円`
  - 東京リージョン想定、監視やバックアップの詳細設定は最小〜標準レベル
- 最小構成（検証〜小規模運用）:
  - 構成例: `EC2 1台 + RDS Single-AZ + S3 + CloudFront（1TB未満）`
  - 目安: `120〜220 USD/月（約18,700〜34,300円/月）`
- 中規模の実運用構成（一般的）:
  - 構成例: `EC2 2台 + RDS Single-AZ〜Multi-AZ + S3 + CloudFront（1〜3TB）+ ALB`
  - 目安: `250〜450 USD/月（約39,000〜70,100円/月）`
- 拡張前提構成（可用性重視）:
  - 構成例: `EC2複数台 + RDS Multi-AZ + CloudFront（3TB超）+ WAF/監視強化`
  - 目安: `500〜900 USD/月（約77,900〜140,200円/月）`
- コスト増の主因:
  - `RDSをMulti-AZにするか`
  - `CloudFrontの転送量（TB/月）`
  - `Webサーバー台数（EC2）`

### 2) microCMSで運用する場合（microCMSホスト + 別フロント）
- microCMS利用料（税抜）:
  - Hobby `0円/月`
  - Team `4,900円/月〜`
  - Business `75,000円/月〜`
- フロント配信費（別途必要）:
  - 目安としてLightsail小規模構成を使うなら `約$5/月〜（約780円/月〜）`
- 実質の月額イメージ:
  - 最小: `0円 + フロント配信費`
  - 実運用（チーム編集）: `4,900円/月〜 + フロント配信費`
- プラン目安（どの利用量でその価格帯か）:
  - Hobby: API無制限、`最大3メンバー`、`最大5 API`、データ転送 `20GB/月`（超過時はAPI停止）
  - Team: `3メンバー`、`10 API`、データ転送 `200GB/月`（超過 `+20円/GB`）
  - Business: `20メンバー`、`30 API`、データ転送 `1TB/月`（超過 `+18円/GB`）
  - 注記: Team/Businessはメンバー数・API数の追加が可能。

### 3) カスタマイズを強めてAWSで運用する場合（管理機能含めて拡張）
- AWS目安（Lightsailのマルチティア例）:
  - Container Service `$7/月（約1,091円/月）`
  - Managed Database `$15/月（約2,337円/月）`
  - Load Balancer `$18/月（約2,805円/月）`
  - 合計 `約$40/月（約6,233円/月）`
- さらに増えやすい費用:
  - 監視、ログ保管、バックアップ、WAF、CI/CD、障害対応の運用工数
- 要点: 月額のインフラ費そのものよりも、保守運用の人件費増加が支配的になりやすい。

## 費用の結論（比較の見方）
- 最小固定費で始めるなら: `WordPress単体` が最も低コストで開始しやすい。
- 表現自由度を上げつつCMSをSaaS化するなら: `microCMS + 別フロント`（ただし月額固定費は上がる）。
- 独自機能を広く持つなら: `AWSカスタム`（インフラ費 + 運用人件費で継続費用が最も重くなる）。

## WordPress単体で進めるデザイン実装の選択肢

## 既存テンプレートを使う
- テンプレート/テーマを選び、色や文字を調整して公開する
- 最短で立ち上げやすい

## 既存テーマをベースにvibeコーディング
- 既存テーマを土台に、必要部分をAI補助で改修する
- 速度とカスタマイズ性のバランスが良い

## デザイン受領後にテーマをvibeコーディング
- デザインカンプや画像を基に、テーマをAI補助で実装する
- 自由度は高いが、詳細再現には仕様の言語化が必要

## なぜ今回はWordPress前提を置くのか
- hands-onで主要パターン（Headless、自作、素実装）を確認済み
- その上で、初期の確実性と運用負荷の観点からWordPress単体が最も安全
- まず運用を成立させてから、必要が出た時点で段階拡張する方がリスクが低い

## テンプレート型CMSの比較（WordPress含む）
- 前提: ここでは「テンプレートで早く立ち上げる」用途を比較する。
- 評価軸:
  - 立ち上げ速度（テンプレ適用のしやすさ）
  - 編集運用のしやすさ
  - 将来の自由カスタマイズへの移行しやすさ

### WordPress.org（自前ホスティング）
- テンプレ運用: しやすい（テーマ/子テーマが豊富）
- 編集運用: しやすい（投稿管理、権限、予約投稿が標準）
- 将来移行性: 高い
  - 理由: PHPテーマ編集、Headless化、別サーバー移管の選択肢が多い
  - 注意: テーマ/プラグイン依存が強いと移行コストは上がる

### Ghost（ホステッド/セルフホスト）
- テンプレ運用: しやすい（メディア向けテーマが多い）
- 編集運用: しやすい（執筆フローがシンプル）
- 将来移行性: 中
  - 理由: コンテンツ移行は比較的しやすいが、独自テーマ/機能再現は再実装が必要

### Webflow
- テンプレ運用: 非常にしやすい（見た目調整が強い）
- 編集運用: 中（運用ルール設計が必要）
- 将来移行性: 中〜低
  - 理由: デザイン再現性は高いが、CMS構造と実装資産の持ち出しに制約が出やすい

### Wix / Squarespace / STUDIO（ノーコード寄り）
- テンプレ運用: 非常にしやすい
- 編集運用: しやすい
- 将来移行性: 低〜中
  - 理由: 初速は速いが、独自実装に寄せる段階で再構築コストが発生しやすい

## 移行しやすさの結論（あとで自由にカスタマイズしたい場合）
- 最初から将来拡張を見込むなら `WordPress.org` が最も扱いやすい。
- ノーコード系は初速に優れるが、後で自由実装に寄せると作り直しが増えやすい。
- したがって本案件（1年で運用開始 + 将来拡張）では、`WordPress.org起点` が最も整合的。

## 補足: Headless移行先としてのSanity
- `Sanity` は、API前提・スキーマコード管理・フロント分離のしやすさから、移行先として有力。
- 特に「将来の柔軟な構造化」や「AIコーディングでスキーマ/フロントを実装する」観点では相性が良い。
- 近年はAI開発界隈でも、コードベースで扱いやすいHeadless CMSとして言及されることが増えている。
- ただし移行時の主要コストは以下:
  - 既存データのマッピング（投稿、カテゴリ、タグ、画像）
  - 本文表現の変換（WordPress HTML/Gutenberg -> 新モデル）
  - プレビュー、権限、公開フローの再設計
- 判断基準:
  - 運用重視（日本語資料/引き継ぎ優先）なら WordPress / microCMS
  - 開発主導で拡張性優先なら Sanity を移行候補に入れる

## 最終判断（現時点）
- 「作る」「続ける」「引き継ぐ」を同時に満たす初手は、`WordPress単体` が最適
