# 生成支援運用候補（比較用）

## 目的
- `evaluation-criteria.md` の評価軸で、生成支援（AI補助）運用候補を比較する。

## 候補一覧（生成支援運用）

| 構成 | 区分 | 向いているケース | メモ |
|---|---|---|---|
| ChatGPT運用 + 人間レビュー | 汎用AI補助 | 記事下書き・要約・タイトル案を高速化したい | レビュー体制を前提に品質を担保しやすい。 |
| Notion AI + CMS転記 | 編集ワークフロー補助 | 企画〜下書きをノート中心で回したい | 編集チーム運用に馴染みやすい。 |
| Jasper + SurferSEO | SEO特化補助 | SEO記事の生産性を上げたい | SEO補助機能が強い。 |
| GitHub Actions + LLM校正 | CI自動化補助 | 公開前に文体・表記ゆれを自動チェックしたい | 技術運用前提で効果が高い。 |

## 先に比較に載せる推奨候補

1. ChatGPT運用 + 人間レビュー
2. Notion AI + CMS転記
3. Jasper + SurferSEO
4. GitHub Actions + LLM校正

## 評価前提

- スコア: `5（非常に良い）- 1（弱い）`
- 想定: ポップカルチャー紹介のブランドメディアを、少人数で継続運用するケース。

## 候補別 採点理由（観点別）

### ChatGPT運用 + 人間レビュー
- 運用継続性 `4`: 日常作業に組み込みやすい。
- 引き継ぎ容易性 `4`: プロンプトとレビュー基準を文書化しやすい。
- 編集体験 `4`: 下書き速度が上がり編集負荷を減らせる。
- 表現自由度 `3`: 最終品質は人間編集で担保する前提。
- 表示性能 `3`: フロント技術には直接影響しない。
- SEO対応力 `3`: 補助可能だが精査が必要。
- セキュリティ運用 `3`: 情報取り扱いルール整備が必要。
- 拡張性 `3`: 運用ルール次第で用途拡大可能。
- プラグイン/拡張エコシステム `2`: 直接拡張より運用設計が中心。
- コスト（初期） `4`: すぐ始められる。
- コスト（運用） `4`: 工数削減効果が出やすい。
- 日本での使いやすさ `5`: 日本語運用しやすい。
- 採用/外注しやすさ `4`: 編集者チームにも展開しやすい。
- ベンダーロックイン `4`: ワークフローを抽象化すれば乗り換え可能。

### Notion AI + CMS転記
- 運用継続性 `3`: 転記フローの手間が残る。
- 引き継ぎ容易性 `4`: ノート中心で運用手順を共有しやすい。
- 編集体験 `4`: 企画から下書きまで一気通貫しやすい。
- 表現自由度 `2`: 最終表現はCMS側の制約を受ける。
- 表示性能 `2`: フロント性能へ直接影響しない。
- SEO対応力 `2`: 専用SEO支援よりは弱い。
- セキュリティ運用 `3`: 情報管理ルール整備が必要。
- 拡張性 `2`: 運用補助としての拡張は限定的。
- プラグイン/拡張エコシステム `2`: 補助用途中心。
- コスト（初期） `5`: 立ち上げが速い。
- コスト（運用） `4`: 編集工数は下げやすい。
- 日本での使いやすさ `5`: 日本語運用と利用者が多い。
- 採用/外注しやすさ `4`: 編集チームに導入しやすい。
- ベンダーロックイン `3`: ノート資産は移行可能だが運用依存は残る。

### Jasper + SurferSEO
- 運用継続性 `4`: SEO記事制作の定常運用に向く。
- 引き継ぎ容易性 `3`: ツール前提の運用設計が必要。
- 編集体験 `4`: 生成とSEO補助を同時に進めやすい。
- 表現自由度 `3`: ブランド文体維持には調整が必要。
- 表示性能 `3`: フロント性能への直接寄与はない。
- SEO対応力 `4`: SEO指標ベースで改善しやすい。
- セキュリティ運用 `4`: SaaS運用前提で管理しやすい。
- 拡張性 `3`: 補助用途としては十分。
- プラグイン/拡張エコシステム `3`: 連携はあるが制約もある。
- コスト（初期） `3`: 導入は中程度。
- コスト（運用） `2`: サブスク費用がかかる。
- 日本での使いやすさ `2`: 日本語運用情報は限定的。
- 採用/外注しやすさ `2`: 実務経験者は多くない。
- ベンダーロックイン `2`: ツール依存が強い。

### GitHub Actions + LLM校正
- 運用継続性 `3`: CI運用としては有効だが技術運用が必要。
- 引き継ぎ容易性 `2`: パイプライン理解が必要。
- 編集体験 `2`: 執筆体験への直接効果は限定的。
- 表現自由度 `4`: チェックロジックを柔軟に設計可能。
- 表示性能 `4`: フロント品質改善の一部に寄与可能。
- SEO対応力 `4`: ルールベースでチェック可能。
- セキュリティ運用 `3`: CIと鍵管理の運用が必要。
- 拡張性 `4`: 自動化範囲を拡張しやすい。
- プラグイン/拡張エコシステム `3`: Action連携が使える。
- コスト（初期） `2`: 初期実装は重い。
- コスト（運用） `3`: 運用コストは中程度。
- 日本での使いやすさ `2`: 技術文書が中心で編集チームには難しい。
- 採用/外注しやすさ `2`: CI/LLM運用人材が必要。
- ベンダーロックイン `4`: OSSベースで移行しやすい。
