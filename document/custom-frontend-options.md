# 自作フロント構成候補（比較用）

## 目的
- `evaluation-criteria.md` の評価軸で、自作フロント構成候補を比較する。

## 候補一覧（自作フロント）

| 構成 | 区分 | 向いているケース | メモ |
|---|---|---|---|
| 素のHTML/CSS/TS（マルチページ） | 非フレームワーク | 小規模サイト、短期公開、機能を絞る運用 | 実装が軽く初動が速い。 |
| Astro Islands + 独自JSON運用 | SSG + 独自運用 | 表現を作り込みつつ、CMSなしで運用したい | 表示速度とUI表現のバランスを取りやすい。 |
| React + Vite（独自実装） | SPA/MPA | UI表現を柔軟に作りたい | 実装自由度は高いが運用設計が必要。 |
| Next.js + 独自管理画面 + Supabase | SSR/SSG + BaaS | 会員機能や投稿機能を将来追加したい | 機能拡張に強いが初期実装が重い。 |
| Nuxt 3 + 独自管理画面 + Firebase | SSR/SSG + BaaS | Vue系で運用したいチーム | 開発しやすいが構成管理が必要。 |
| SvelteKit + PocketBase | SSR/SSG + 軽量BaaS | 小〜中規模で軽快に作りたい | 生産性は高いが国内知見が少ない。 |

## 先に比較に載せる推奨候補

1. 素のHTML/CSS/TS（マルチページ）
2. Astro Islands + 独自JSON運用
3. React + Vite（独自実装）
4. Next.js + 独自管理画面 + Supabase
5. Nuxt 3 + 独自管理画面 + Firebase
6. SvelteKit + PocketBase

## 評価前提

- スコア: `5（非常に良い）- 1（弱い）`
- 想定: ポップカルチャー紹介のブランドメディアを、少人数で継続運用するケース。

## 候補別 採点理由（観点別）

### 素のHTML/CSS/TS（マルチページ）
- 運用継続性 `3`: 構成は単純だが、更新量が増えると手作業が増えやすい。
- 引き継ぎ容易性 `3`: 設計が明快なら引き継げるが、規約がないと属人化しやすい。
- 編集体験 `2`: 非エンジニア向けの編集UIがなく、更新ハードルが高い。
- 表現自由度 `5`: フロント実装の制約が少なく自由に作れる。
- 表示性能 `5`: 軽量構成で高い表示速度を出しやすい。
- SEO対応力 `3`: 実装は可能だが運用での入力漏れが起きやすい。
- セキュリティ運用 `4`: 動的機能が少なければ安全性を保ちやすい。
- 拡張性 `2`: 機能追加時にゼロから実装が増えやすい。
- プラグイン/拡張エコシステム `2`: CMS型の即導入資産は少ない。
- コスト（初期） `5`: 最小構成なら低コストで開始できる。
- コスト（運用） `5`: インフラ費を低く保ちやすい。
- 日本での使いやすさ `4`: 基本技術として情報が多く理解しやすい。
- 採用/外注しやすさ `4`: フロント実装者を確保しやすい。
- ベンダーロックイン `5`: 特定サービス依存が少なく可搬性が高い。

### Astro Islands + 独自JSON運用
- 運用継続性 `4`: 構成を整理すれば安定運用しやすい。
- 引き継ぎ容易性 `3`: ルールがあれば引き継げるが、独自運用設計の理解が必要。
- 編集体験 `2`: JSON/ファイル更新中心で非エンジニアには不向き。
- 表現自由度 `5`: コンポーネント設計で高い表現力を持てる。
- 表示性能 `5`: 静的配信中心で高速化しやすい。
- SEO対応力 `3`: 実装で対応可能だが、運用入力のガイド設計が必要。
- セキュリティ運用 `4`: 動的面を減らせるため保守しやすい。
- 拡張性 `3`: 拡張は可能だが、独自実装コストはかかる。
- プラグイン/拡張エコシステム `3`: フロント拡張は可能だが運用機能は自作寄り。
- コスト（初期） `4`: 比較的短期間で構築しやすい。
- コスト（運用） `4`: 静的運用でランニングを抑えやすい。
- 日本での使いやすさ `3`: 技術情報は増えているが運用事例は限定的。
- 採用/外注しやすさ `3`: 対応できる人材はいるが母数はReactより少なめ。
- ベンダーロックイン `4`: データ運用を分離すれば移行しやすい。

### React + Vite（独自実装）
- 運用継続性 `3`: 実装自由度は高いが、運用機能の自作管理が必要。
- 引き継ぎ容易性 `3`: 設計と規約を整えないと属人化しやすい。
- 編集体験 `2`: 編集管理画面がないと更新は技術者依存になる。
- 表現自由度 `5`: UI表現は自由に実装できる。
- 表示性能 `4`: 最適化次第で高性能だが、設計品質に依存。
- SEO対応力 `3`: 実装可能だが、構成次第で対策難易度が変わる。
- セキュリティ運用 `3`: API連携や依存更新の管理が必要。
- 拡張性 `4`: 要件追加には対応しやすい。
- プラグイン/拡張エコシステム `3`: ライブラリは豊富だが統合作業が必要。
- コスト（初期） `3`: 最低限は作りやすいが、運用機能を入れると増える。
- コスト（運用） `3`: 継続的な開発保守が必要。
- 日本での使いやすさ `3`: 情報は多いが運用テンプレートは少ない。
- 採用/外注しやすさ `4`: React人材の母数は多い。
- ベンダーロックイン `4`: 技術依存はあるが移行余地は高い。

### Next.js + 独自管理画面 + Supabase
- 運用継続性 `3`: 機能豊富だが、構成が複雑になりやすい。
- 引き継ぎ容易性 `2`: フロント/バック双方の理解が必要で引き継ぎ難度が上がる。
- 編集体験 `3`: 管理画面を作り込めば改善できるが初期は粗くなりやすい。
- 表現自由度 `5`: フロントの自由度が高くブランド表現に強い。
- 表示性能 `4`: 適切な設計で高性能を出せる。
- SEO対応力 `4`: 実装で細かく制御しやすい。
- セキュリティ運用 `3`: 認証/DB/API運用の責務が増える。
- 拡張性 `5`: 将来機能に対応しやすい。
- プラグイン/拡張エコシステム `3`: ライブラリ活用は可能だが統合負荷がある。
- コスト（初期） `2`: 初期実装範囲が広く工数が重い。
- コスト（運用） `2`: 継続保守コストが上がりやすい。
- 日本での使いやすさ `3`: 技術情報は多いが総合運用の事例設計が必要。
- 採用/外注しやすさ `3`: フルスタック対応人材の確保が必要。
- ベンダーロックイン `3`: BaaS依存はあるが移行余地は残る。

### Nuxt 3 + 独自管理画面 + Firebase
- 運用継続性 `3`: 構成を整理すれば運用可能だが、管理項目は多い。
- 引き継ぎ容易性 `2`: 独自実装部分が増えるほど引き継ぎ難度が上がる。
- 編集体験 `3`: 管理画面品質に依存し、運用初期は改善が必要。
- 表現自由度 `4`: 表現は十分作り込める。
- 表示性能 `4`: 構成次第で高性能化できる。
- SEO対応力 `3`: 実装で対応可能だが、SSR/SSG設計を要する。
- セキュリティ運用 `3`: Firebase設定・権限設計の運用が必要。
- 拡張性 `4`: 機能追加しやすい構成。
- プラグイン/拡張エコシステム `3`: 周辺ライブラリはあるが運用機能は自作が多い。
- コスト（初期） `3`: 初期構築工数は中程度。
- コスト（運用） `3`: サービス利用料と保守が継続する。
- 日本での使いやすさ `2`: 国内事例はあるが母数は多くない。
- 採用/外注しやすさ `3`: Vue人材は確保可能だが限定的。
- ベンダーロックイン `3`: Firebase依存の設計だと移行負荷がある。

### SvelteKit + PocketBase
- 運用継続性 `3`: 軽量に作れるが、運用ノウハウの蓄積が必要。
- 引き継ぎ容易性 `2`: 対応経験者が少なく、引き継ぎ難度が高め。
- 編集体験 `3`: 管理画面次第で改善可能だが初期品質に差が出る。
- 表現自由度 `4`: UI実装の自由度は十分高い。
- 表示性能 `4`: 軽量構成で高速化しやすい。
- SEO対応力 `3`: 実装で担保できるが設計依存。
- セキュリティ運用 `3`: 自前運用要素の管理が必要。
- 拡張性 `4`: 中規模までの拡張は行いやすい。
- プラグイン/拡張エコシステム `2`: 実務向け資産はReact系より少ない。
- コスト（初期） `3`: 小規模開始はしやすい。
- コスト（運用） `3`: 運用体制次第で変動しやすい。
- 日本での使いやすさ `2`: 日本語情報・国内事例が少ない。
- 採用/外注しやすさ `2`: 人材母数が少なく外注先も限られる。
- ベンダーロックイン `4`: 特定SaaS依存を避けやすい。
